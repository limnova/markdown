## Layout And Visual Rules

- Evolve the Phase 1 centered panel into a full-height two-pane desktop shell instead of introducing a new visual system.
- Use a fixed 304px left sidebar and a flexible right document pane. Both sit on the same warm neutral background and keep the existing border radius, border tone, and soft shadow vocabulary from Phase 1.
- Keep the sidebar visually calm: top workspace header, then tree actions, then the file tree. Do not add IDE-like tab bars, dense tool ribbons, or multi-panel nesting.
- The document pane header must always show the current document path on the left and a dedicated save-status area on the right. Save trust must stay visible without opening a modal or toast history.
- The ready-shell reading order is fixed: document-pane empty state first, `New note` CTA second, save-status area third, and tree chrome last.
- Tree rows use 40px minimum height, 14px semibold primary labels, 14px muted path/meta text, and a 16px indentation step per depth level.
- Active Markdown rows use an accent-tinted background, a 1px accent outline, and stronger text weight. Hover states on non-active rows remain neutral and low-motion.
- Folder rows and Markdown rows look first-class. Non-Markdown rows remain visible for orientation but are visually weakened: muted label, low-contrast badge, no active treatment, no accent hover.
- Inline create and rename states reuse the same row frame as normal tree rows. The input is embedded in-row, auto-focused, and never shown as a separate naming modal.
- Valid drag destinations show a 4px accent indicator band and a soft inset highlight. Invalid destinations show no highlight and must not fake acceptance.
- Destructive confirmations use the same shell language: warm surface, low-contrast border, left-aligned copy, and one clearly destructive primary action. No bright-red full-screen takeover.

## Save Status Semantics

| State | Visual treatment | Copy |
|-------|------------------|------|
| Unsaved | Warm amber chip: background `#F5E4BF`, text `#8A5A00` | Unsaved |
| Saving | Cool blue chip: background `#E4EEFF`, text `#1F57D8` | Saving |
| Saved | Calm green chip: background `#E6F3EA`, text `#2F6B3E` | Saved |
| Save failed | Destructive chip: background `#F7E3DE`, text `#C34A36` | Save failed |

- The save-status area is always present in the document header, even when the current state is stable.
- Show the state chip plus a short trailing text line when useful, such as the current file name while saving or the last successful save time after save completes.
- `Saved` must not flash and disappear instantly. Hold the success state long enough to be readable, then keep the area occupied with a quiet saved/timestamp treatment.
- `Save failed` blocks routine transitions and stays visible until the next successful save or the user cancels the pending navigation.

## Interaction Contract

- The root workspace stays the only content scope. The tree always renders within that boundary and never exposes out-of-root move targets.
- On entry to the ready state, the workspace root is expanded by default. Any ancestor chain of the current document also stays expanded so the active note is always locatable in the tree.
- Clicking a folder toggles expand/collapse in place. Clicking a Markdown file opens it in the current document pane.
- Clicking a non-Markdown file stays in-app, does not hand off to the OS, and does not replace the current document. Show inline feedback that the file type is not supported for opening here yet.
- The primary tree creation action is `New note`. A secondary tree action may create a folder, but it must remain visually quieter than the note action.
- Creating a note or folder inserts a pending-name row inline in the current tree context immediately. Do not open a naming modal first.
- If the user abandons or submits an empty create row, the app auto-assigns a collision-safe default name: `Untitled.md`, `Untitled 2.md`, `Untitled 3.md` for notes; `New Folder`, `New Folder 2`, `New Folder 3` for folders.
- A new Markdown file opens immediately after creation and places focus in the document pane so writing can start without an extra click.
- Rename happens inline inside the tree row, not in a modal. The row frame stays stable so the tree does not jump while editing the name.
- Drag-and-drop supports workspace reorganization. Valid move targets are folder rows and the root-level drop zone; unsupported files and non-target rows are never shown as valid destinations.
- If the current open document is renamed or moved, the editor session remains open and follows the item seamlessly. Update the tree highlight, document path label, and save-status context without closing the document.
- `Ctrl+S` saves the current document immediately. Successful save updates the save-status area only; do not use an extra success modal or toast.
- Routine transitions that could abandon unsaved work, including switching to another file or leaving the workspace view, auto-save first and continue only after save succeeds.
- If routine auto-save fails, block the transition, keep the current document in place, and surface the failure in the persistent save-status area plus supporting inline error text.
- Deleting any file or folder always requires explicit confirmation before the operation runs.
- Delete confirmation prefers Recycle Bin wording and behavior when the platform allows it. Do not describe deletion as permanent unless fallback behavior makes that necessary.
- Deleting the current dirty note is a special case: the confirmation primary action must preserve work first, using `Save and move to Recycle Bin`, with `Keep note` as the only secondary path. Do not offer a casual discard-first shortcut in this phase.

## State Inventory

| State | Required UI treatment |
|-------|------------------------|
| Ready shell, no document selected | Sidebar tree visible; document pane shows a calm empty state with `Create your first note` copy and the primary `New note` action |
| Workspace has folders but no Markdown files yet | Sidebar remains interactive; document pane uses the same empty-state contract and suggests creating a note or folder |
| Folder expanded | Chevron rotated/open, children indented beneath parent, no extra card wrapper |
| Folder collapsed | Children hidden, parent row remains stable in height |
| Inline note creation | Pending row inserted in tree with focused text input and Markdown note affordance |
| Inline folder creation | Pending row inserted in tree with focused text input and folder affordance |
| Unsupported non-Markdown click | Current document remains unchanged; inline banner explains the file type cannot be opened here yet |
| Current document unsaved | Persistent amber `Unsaved` chip in the document header |
| Current document saving | Persistent blue `Saving` chip; navigation waits for completion |
| Current document saved | Persistent green `Saved` chip or saved timestamp treatment in the same header area |
| Save failed | Persistent red `Save failed` chip plus inline supporting copy; routine transitions blocked |
| Valid drag target hovered | Accent drop indicator and inset highlight on the destination folder or root drop zone |
| Invalid drag target hovered | No acceptance highlight; cursor/treatment stays non-affirming |
| Delete clean item | Confirmation surface with `Move to Recycle Bin` primary action and `Keep item` secondary action |
| Delete current dirty note | Confirmation surface with `Save and move to Recycle Bin` primary action and `Keep note` secondary action |
| Current document renamed or moved | Session remains open; path label, tree highlight, and save-status context update without closing the editor |
