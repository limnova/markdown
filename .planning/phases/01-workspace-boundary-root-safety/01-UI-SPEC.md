## Layout And Visual Rules

- Replace the starter logos and link clutter entirely. Phase 1 must look like a desktop writing tool, not a scaffold demo.
- Use one quiet full-height canvas with a single centered panel between 520px and 560px wide.
- Keep all welcome and invalid-workspace content left-aligned inside the panel even though the panel itself is centered in the window.
- Use a 1px low-contrast border plus a soft shadow on the panel. Avoid glossy gradients, glassmorphism, or marketing-style hero art.
- Keep the primary CTA visually dominant. Any secondary action must be text-only and visually quieter than the primary button.
- Show the current workspace path in a compact pill or inset row. Long Windows paths must truncate in the middle, not only at the end.
- External-link rows must read as informational, not actionable: muted text, reduced contrast, no hover lift, no selected state, no chevron.

---

## Interaction Contract

- First launch opens directly into a dedicated welcome state with one primary action: choose the workspace directory.
- The welcome state is a single-step surface. Do not add onboarding progress, feature tours, recent-file carousels, or setup checklists.
- App relaunch attempts to restore the saved workspace automatically. Do not ask for confirmation if the saved path is still valid.
- If restore validation takes longer than a brief frame, show a compact neutral loading message in the same panel skeleton instead of a blank flash.
- If the remembered path is missing or inaccessible, replace the shell with a blocked invalid-workspace state. Do not silently clear the saved path and do not drop the user back to a generic empty home.
- The invalid-workspace state must show the broken path, explain that the folder is unavailable, and keep one primary reselect action visible at all times.
- Any out-of-root access attempt must fail non-destructively with an inline banner or top-of-shell alert. Do not use a confirmation modal for rejection-only cases.
- Rejection copy must name the rule, not blame the user. Preferred pattern: "That location is outside the current workspace and can't be opened here."
- If this phase renders any root contents list, entries that resolve outside the root appear as external-link rows only. They may be visible for awareness but are never expanded, opened, edited, searched, or chosen as operation targets.
- Boundary errors must not clear the current root, discard state, or navigate away from the current shell.

## State Inventory

| State | Required UI treatment |
|-------|------------------------|
| First launch / no workspace | Centered welcome panel with display title, one body paragraph, one primary CTA |
| Restoring saved workspace | Same panel frame with compact loading copy; no spinner-heavy screen takeover |
| Valid workspace restored | Shell header shows current root path and boundary-safe status; no modal confirmation |
| Saved workspace invalid | Blocked panel with destructive emphasis, broken path display, one primary reselect CTA |
| Out-of-root rejection | Inline banner in the active shell; current selection and workspace remain intact |
| External-link node encountered | Disabled informational row with external-link icon/badge and no interaction affordances |
